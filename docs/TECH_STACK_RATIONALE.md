## 1. 为什么是 Qwen-Omni

### 1.1 传统管线 vs Qwen-Omni

传统多模态 AI 产品（摄像头 + 语音对话）的管线：

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 摄像头   │ → │ 视觉模型 │ → │ 语音识别 │ → │  LLM    │ → │  TTS    │ → 音箱
│ 采集帧   │   │ YOLO/CLIP│   │ Whisper │   │ GPT-4o  │   │ ElevenLabs│
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
    50ms           200ms          300ms         1500ms         800ms
                                  ═══════════════════════════════════
                                  累计延迟: ~2.85s (理想情况)
                                  实际: 3-5s (网络 + 排队 + 调度)
                                  错误: 5 个环节级联放大错误率
```

Qwen-Omni 管线：

```
┌──────────────────┐          ┌──────────────────┐          ┌─────────┐
│  摄像头 + 麦克风  │ ───────→ │ Qwen-Omni (单模型)│ ───────→ │  音箱    │
│  画面 + 音频输入  │          │ 视觉+音频→文本+语音│          │ 语音播放 │
└──────────────────┘          └──────────────────┘          └─────────┘
                                      < 1.5s (本地)
                                      单一推理，无级联错误
```

### 1.2 Qwen-Omni 架构优势

| 特性 | 传统方案 | Qwen-Omni |
|------|---------|-----------|
| **模型数量** | 4-5 个独立模型 | 1 个模型端到端 |
| **延迟** | 3-5 秒 | < 1.5 秒（本地）/ 1-3 秒（云端） |
| **错误级联** | 每个环节出错 → 下游放大 | 无级联 |
| **儿童语音** | ASR 对儿童英语识别率低（~60-70%） | 原生音频理解，绕过文字转写瓶颈 |
| **端到端优化** | 各部分独立优化，难以联合调优 | 单一模型，可针对场景微调 |
| **部署复杂度** | 5 个服务 5 套运维 | 1 个推理服务 |
| **多模态理解** | 视觉 → 文本 → LLM，丢失空间/时序信息 | 原生视觉理解（Thinker-Talker 架构） |

### 1.3 Qwen-Omni 关键能力（对我们的场景）

| 能力 | 说明 | 对应场景 |
|------|------|---------|
| **图片理解** | 输入图片，输出文本/语音描述 | 看到孩子拿玩具 → 用英语描述 |
| **视频理解** | 连续帧输入，理解动作和活动 | 识别孩子在画画 vs 玩玩具 vs 看绘本 |
| **音频理解** | 直接理解语音语义（非 ASR 转写） | 孩子说英语（即使发音不准）→ 理解意图 |
| **语音输出** | 原生流式语音生成（无需 TTS） | 自然流畅的英语语音回复 |
| **文本输出** | 同时生成文本和语音 | 语音给孩子听 + 文本给家长报告 |
| **双语** | 中英文混合训练（Qwen 系列独特优势） | 理解中英夹杂的儿童语言 |
| **MoE 架构** | 推理时只激活部分参数 | 7B 激活 ~2B，30B 激活 ~8B，更快推理 |

### 1.4 Qwen-Omni 两个版本

| | Qwen2.5-Omni-7B | Qwen3-Omni-30B |
|---|---|---|
| **总参数** | 7B (MoE) | 30B (MoE) |
| **激活参数** | ~2B | ~8B |
| **视觉编码器** | Qwen2.5-VL | Qwen3-VL |
| **音频编码器** | Whisper-style | 原生音频 tokenizer |
| **语音输出** | 流式语音 | 更自然的语音（更多训练数据） |
| **英语质量** | 良好 | 优秀（接近 native-like） |
| **推理设备** | Jetson Orin AGX (64GB) | 需 A100/H100 云端 |
| **推理延迟(本地)** | < 1.5s | 不可本地部署（需云端） |
| **推理延迟(云端)** | N/A | 1-3s |
| **成本** | 电费 ~¥20/月 | API ~¥300/月 |

### 2.2 排除原因总结

| 模型 | 排除原因 |
|------|---------|
| **GPT-4o** | 闭源、不可本地部署、API 成本高（视觉+音频 token 贵）、延迟不可控 |
| **Gemini 2.0 Flash** | 无原生语音输出（需拼接 TTS）、不可本地部署 |
| **Claude 3.5 Sonnet** | 无音频能力、不可本地部署 |
| **Qwen-VL-Max** | 无音频能力 |
| **Llama 4** | 无音频能力、无中文 |
| **MiniCPM-o** | 7-8B Dense 架构（非 MoE），推理速度不如 MoE，语音质量不如 Qwen-Omni |

### 2.3 Qwen-Omni 的独特优势

1. **唯一全模态开源 MoE 模型**：视觉 + 音频输入 + 语音输出 + 文本输出，全部原生
2. **中英双语原生训练**：Qwen 系列的训练数据包含大量高质量中英文数据，天然理解中国孩子中英夹杂的语言
3. **MoE 架构低延迟**：推理时只需激活 25-30% 参数，7B 模型等效推理成本约 2B Dense 模型
4. **可本地部署**：7B MoE 在 Jetson Orin 64GB 上可部署（需 TensorRT-LLM 量化优化）
5. **阿里巴巴背书**：Qwen 系列持续迭代，有强大的社区和商业支持

### 2.4 Thinker-Talker 架构（Qwen-Omni 核心创新）

```
输入: 图片/视频帧 + 音频流
        │
        ▼
┌───────────────────────────┐
│        Thinker             │  ← 理解阶段: 把看到和听到的融合为统一表示
│  (多模态编码 + 交叉注意力)  │
└───────────┬───────────────┘
            │
            ▼
┌───────────────────────────┐
│        Talker              │  ← 生成阶段: 同时输出文本和语音 token
│  (双轨解码: 文本 + 语音)     │
└───────────┬───────────────┘
            │
    ┌───────┴───────┐
    ▼               ▼
  文本输出        语音输出
  (家长报告)      (孩子听到)
```

**对我们场景的意义**：
- Thinker 阶段识别：孩子拿的什么、在做什么、情绪状态
- Talker 阶段同时产生：给孩子听的语音 + 给家长看的学习记录文本

### 3.2 推荐方案：Jetson AGX Orin 64GB（开发阶段）

| 维度 | 评估 |
|------|------|
| **为什么不是 Orin NX 16GB** | Qwen2.5-Omni-7B 在 INT4 量化后约需 8-12GB 显存，NX 16GB 刚好够但有风险。AGX 64GB 留足余量，可同时跑视觉预处理、VAD、推理。 |
| **为什么不是 Orin Nano** | 40 TOPS 不足以在 < 2s 内完成 7B MoE 模型的推理 |
| **为什么不是 Intel NUC** | 无 CUDA/TensorRT 生态，Qwen-Omni 的 GPU 推理依赖 CUDA |
| **为什么不是 Raspberry Pi** | 算力差两个数量级，Hailo-8L 只能跑轻量 YOLO，不能跑 LLM |
| **量产降本路径** | 开发用 AGX Orin，量产后可选 Orin NX 16GB (优化后的量化模型) 或等待下一代 Orin Nano (2026 预期大幅升级) |

```

              ┌─────────────────┼─────────────────┐

    │ 适合:              │              │ 适合:              │
    │ • 物品识别与描述    │              │ • 绘本精读与讨论     │
    │ • 简单对话         │              │ • 复杂纠音反馈     │
    │ • 游戏引导         │              │ • 家长报告生成     │
    │ • 状态检测         │              │ • 长故事创作        │
    │ • 注意力/情绪判断  │              │ • 开放域问答        │
    │                   │              │                   │
    │ 延迟: < 1.5s       │              │ 延迟: 1-3s         │

    │ 隐私: 数据不出设备  │              │ 隐私: 需上传图片/音频 │
    └──────────────────┘              └──────────────────┘
```

### 4.2 负载分配策略

| 交互类型 | 路由 | 理由 |
|---------|------|------|
| 物体/活动识别 | 本地 | 高频、低延迟要求、隐私敏感（画面不离开设备） |
| 简单对话（≤ 3 轮） | 本地 | 低延迟让对话自然 |
| 绘本共读 | 云端 | 需要更强推理能力（理解故事情节） |
| 复杂纠音 | 云端 | 30B 的英语语音理解更准确 |
| 游戏互动 | 本地 | 低延迟关键（游戏节奏快） |
| 注意力/情绪检测 | 本地 | 高频、隐私极度敏感 |
| 家长每日报告 | 云端 | 非实时，可批量处理 |
| 长故事创作 | 云端 | 需要更强创造力 |

### 4.3 降级策略

```
正常运行:  WiFi 连接 → 本地 + 云端混合
          │
WiFi 断开:  自动切换为纯本地模式
          │  LED 黄色闪烁一次（告知模式切换）
          │  复杂功能降级：绘本精读 → 基础描述
          │                  长故事 → 短句旁白
          │
WiFi 恢复:  自动恢复混合模式
          │  LED 蓝色呼吸（正常）
```

## 5.5 语音输出与打断架构

### 实时可打断 vs ChatGPT

ChatGPT Advanced Voice Mode 能做到"实时可打断"是因为 GPT-4o 是端到端多模态模型——音频连续输入，模型语义理解用户在打断，响应侧流式输出。Qwen-Omni **架构上是同代的**（Thinker-Talker），理论上有同样的能力。当前限制是API调用模式（请求-响应），不是模型能力。

### 我们的三层方案

```
Level 1 [当前 — VAD打断]: 播放线程独立运行, VAD同时监听.
  孩子出声 → PlaybackController.interrupt() → 立即停止播放+丢弃队列.
  粒度: 50ms chunks. 误触发风险: 咳嗽/噪音 → 用speech_confidence>0.5过滤.

Level 2 [短期 — 流式响应]: Qwen-Omni流式输出音频token,
  逐chunk推入PlaybackController → 首词延迟<500ms.
  仍在播放中运行VAD → 可打断.

Level 3 [未来 — 端到端双工]: Qwen-Omni持续接收音频流,
  模型语义理解打断意图 → 自己停止. ChatGPT级体验.
  需要Qwen-Omni API支持streaming-duplex(暂未公开).
```

**关键实现**：`audio_io.py` — `PlaybackController` 类

- 独立播放线程，50ms chunk粒度
- VAD **永远在线**（Emma说话期间也一样）
- 检测到孩子声音 → 立即中断 + 丢弃剩余音频
- 不需要录音按钮 —— 全程自然对话

### 语音合成方案

| 方案 | 音质 | 可打断 | 自带嘴型时序 | 许可证 |
|------|------|--------|------------|--------|
| **Qwen-Omni Talker** | ⭐⭐⭐⭐ | ✅ 流式 | ❌ 需whisper后处理 | Apache 2.0 |
| **Kokoro-82M + sherpa-onnx** | ⭐⭐⭐⭐⭐ | ✅ 独立播放线程 | ❌ 需whisper后处理 | Apache 2.0 |
| FastSpeech2 + HiFi-GAN | ⭐⭐⭐ | ✅ | ✅ 原生duration | Apache 2.0 |

**推荐**: Qwen-Omni Talker（主）+ Kokoro（备选，音质更好且Apache 2.0无风险）。

### 嘴型同步管线

```
Qwen-Omni/Kokoro 生成音频
        ↓
whisper.cpp word-level timestamps
  {word: "apple", start_ms: 200, end_ms: 450}
        ↓
avatar.py PHONEME_TO_VISEME 查表
  "apple" → /ae/ → Viseme.AA
        ↓
avatar.py EmmaAvatar 渲染对应嘴型SVG/Live2D
```

**延迟分析**：TTS生成(<500ms) + whisper对齐(~50ms) + 查表+渲染(<5ms) = ~550ms总延迟。
